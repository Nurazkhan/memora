Think of Memora not as a single program, but as a carefully orchestrated system where each part has a very strict responsibility. The reason many applications of this kind feel slow or unreliable is because they mix heavy computation, UI rendering, and data storage into one layer. Your goal is to keep these concerns completely separated so that the user interface always remains responsive, regardless of how much work is happening behind the scenes.

When the application starts, Electron launches a local environment that includes the React interface and a background Python service exposed through a local API. From the user’s perspective, this feels like one application, but internally it behaves like a small distributed system running on the same machine. The frontend never directly processes images or runs algorithms. Instead, it sends requests to the backend, which is responsible for all logic and computation. This separation is what allows you to maintain smoothness in the interface even when processing thousands of images.

When a new project is created, the system initializes both a database and a structured file system. The database, typically SQLite at this stage, stores only lightweight metadata such as references to images, detected faces, cluster assignments, and relationships between people. The actual image files are stored on disk in a well-organized directory structure. This distinction is critical because databases are not optimized for storing large binary data like images, and attempting to do so would significantly degrade performance. By keeping images on disk and metadata in the database, you ensure that queries remain fast and memory usage stays predictable.

As soon as images are added to the project, they are not processed immediately in the main thread. Instead, they are placed into a background processing queue. This queue is handled by worker processes that operate independently of the user interface. The very first operation performed on each image is the generation of a thumbnail. This step happens quickly and produces a much smaller version of the image, typically a few hundred pixels in size. The importance of this cannot be overstated: from this point onward, the user interface will only ever display these thumbnails. Full-resolution images are reserved strictly for export. By making this decision early, you eliminate the primary cause of lag in image-heavy applications.

After the thumbnail is generated, the system proceeds with face detection and embedding extraction. Each detected face is converted into a numerical vector that represents its unique features. These vectors are the foundation of the entire recognition system. Rather than storing these vectors directly in the database, which could become inefficient, you store them in a compact binary format on disk and keep references in the database. This approach minimizes database load while still allowing fast access when needed.

Once enough embeddings have been collected, the clustering process begins. This process groups similar face vectors together, effectively identifying all images that belong to the same person. The clustering runs entirely in the background and does not block the interface. When it completes, the system has formed groups of faces, each corresponding to a potential individual. At this stage, the system does not yet know the names of these individuals; it only knows that certain faces are likely the same person.

The next phase is the tagging interface, which is designed to minimize user effort. Instead of showing all images, the system selects a small number of representative thumbnails for each cluster and sends only those to the frontend. This keeps data transfer light and rendering fast. The user assigns a name to the cluster, and this assignment propagates to all associated faces. This is where the system transitions from raw computation to meaningful organization, and it is also where a large amount of manual work is eliminated.

After identities are assigned, the system builds a relational structure that maps which students appear together in each photo. This is done by iterating over the metadata rather than the images themselves, making it computationally inexpensive. The result is a co-occurrence model that captures relationships between individuals. This model becomes the basis for selecting group photos later, as it allows the system to identify which students frequently appear together.

The next stage involves evaluating the quality of individual images. For each photo associated with a student, the system computes several metrics. Sharpness is measured using a Laplacian-based method, which provides a quick estimate of image clarity. Face size is determined from the bounding box, ensuring that images where the subject is clearly visible are prioritized. Pose is inferred from facial landmarks, allowing the system to distinguish between frontal and angled shots. Expression can be approximated using simple classifiers or landmark-based heuristics. These metrics are combined into a scoring function that ranks images by overall quality.

However, ranking alone is not sufficient, because it can lead to selecting multiple nearly identical images. To address this, the system introduces a diversity constraint. Each image is categorized by pose or expression, and the final selection ensures variation by choosing the best image from different categories. This results in a more natural and visually appealing set of photos, which is particularly important for album presentation.

Templates are handled as structured data rather than visual documents. Each template defines pages and frames, with each frame assigned a type such as individual or group. When a template is loaded, the frontend renders it visually while the backend interprets it logically. This dual representation allows the system to automate placement while still giving the user a clear visual understanding of the layout.

When the auto-fill process runs, the system combines all previously computed data. For each frame, it selects appropriate images based on type and ranking. Individual frames receive the best-ranked portraits, while group frames are filled using images that contain strongly connected individuals according to the co-occurrence model. This process is deterministic and fast because all heavy computation has already been completed earlier.

The user interface at this stage must remain extremely responsive, and this is achieved through careful rendering strategies. Instead of rendering all images at once, the interface uses virtualization techniques so that only visible elements are actually drawn. As the user scrolls, new elements are rendered and old ones are removed. Combined with thumbnail usage and memory caching, this ensures that even large datasets feel smooth. Frequently accessed images are kept in memory using a least-recently-used strategy, allowing instant retrieval without repeated disk access.

When the user interacts with a specific frame, the system does not fetch all possible alternatives. Instead, it queries the backend for a small, pre-ranked subset of relevant images. This keeps interactions fast and avoids overwhelming the user with unnecessary options. The editing experience feels immediate because all heavy decisions have already been made in advance.

Finally, when the album is exported, the system switches to using full-resolution images. At this point, performance is less critical because the process is expected to take some time. The rendering engine composes each page according to the template, places images precisely, and generates a high-quality PDF suitable for printing. Because this is done as a separate step, it does not affect the responsiveness of the main application.

The overall smoothness of Memora comes from a single guiding principle: the user interface never performs heavy work and never waits for it. All expensive operations are executed ahead of time, cached, and reused. The frontend deals only with lightweight representations and precomputed results. As long as this architecture is respected, the application will feel fast and reliable, even when handling large volumes of data.
